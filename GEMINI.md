# github-cicd Project Context (GEMINI.md)

このファイルは、`github-cicd` リポジトリにおいて Antigravity が遵守すべきプロジェクト固有の指針です。グローバルな `GEMINI.md` の設定を補完・強化します。

## 1. プロジェクトの目的
このリポジトリは GitHub Actions の CI/CD パイプラインにおけるベストプラクティス、セキュリティ、および実装例のコレクションです。新しいワークフローの追加や既存コードの修正は、常に「教育的で安全な手本」であることを意識して行います。

## 2. セキュリティ・実装の絶対ルール
このリポジトリのワークフローおよびコード変更においては、以下のルールを厳格に適用します。

### GitHub Actions セキュリティ
- **中間環境変数の利用**: コンテキスト（`${{ github.event... }}` 等）は `run` ステップ内で直接使用せず、必ず `env` ブロックで中間環境変数に代入してから使用します。
- **権限 (Permissions) の最小化**:
    - ワークフローレベルでは必ず `permissions: {}` を明示し、デフォルトの書き込み権限を無効化します。
    - 必要な権限は各ジョブレベルで最小限に定義します。
- **Action の SHA 固定**: `uses` で使用する Action は、タグ（`v3` 等）ではなく完全なコミット SHA（40文字）で指定し、コメントでバージョンを併記します。
- **タイムアウトの設定**: 全てのジョブに `timeout-minutes`（推奨: 5〜10分）を設定し、ハングを防止します。

### コード品質と検証
- **検証の自動化**: ワークフローを変更した際は、必ず `actionlint` および `conftest` (Rego ポリシー) による検証をパスすることを確認します。
- **Go コード**: 新しい機能や修正には必ずテストコードを伴い、`go test ./...` で検証します。
- **シェルスクリプト**: `set -euo pipefail` を含め、エラーに対して堅牢な実装を心がけます。

## 3. ディレクトリ構造と命名規則
- `.github/workflows/`: ワークフローファイル名は、既存の慣習に合わせ **スネークケース (`snake_case.yml`)** を推奨します。
- `go/`: Go プロジェクトは標準的なディレクトリ構造を維持し、`go.mod` による依存関係管理を徹底します。
- `policy/`: ワークフローの静的解析用 Rego ポリシーを配置し、変更時は `conftest` で整合性を確認します。

## 4. 開発サイクル
1. **リサーチ**: 既存の類似ワークフローや `README.md` の記述を確認し、整合性を保ちます。
2. **実装**: 上記のセキュリティルールを全て適用したコードを記述します。
3. **検証**: `actionlint` や `conftest` を実行し、品質を担保します。
4. **ドキュメント**: 新しいワークフローや重要な変更を加えた場合は、必要に応じて `README.md` に追記します。
